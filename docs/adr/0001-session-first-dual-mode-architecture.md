# ADR 0001：以创作会话为核心的双模式架构

日期：2026-05-12

## 状态

提议中

## 背景

TBird 最初接近 Chatbox 式写作工具，但现在产品方向已经转向“圆桌小说盒子”：一个个人写作工作间。用户既可以直接和写手创作，也可以打开会议式圆桌，让多个 AI 议员参与讨论、批评、审批、建议，再由写手继续生成正文。

当前实现仍然有大量逻辑集中在 `src/main.js` 里，包括 UI 渲染、AI 调用、会话状态、圆桌流程、设置面板、Android 行为等。

新的产品方向需要更清楚的模型来支持：

- 直接和写手交流
- 圆桌共创
- 共享小说资产
- 单个参与者独立 API 或模型设置
- 议员默认短发言
- 写手正文输出作为特殊消息
- Android 移动端优先验证

## 决策

采用以创作会话为核心的架构。

一个创作会话是最高级创作单位。交流模式和圆桌模式不是两套系统，而是同一个创作会话的两种视图。

创作会话拥有：

- 小说资产
- 参与者
- 消息记录
- 模式状态
- 外观设置
- 会话设置

写手是两种模式共享的特殊参与者。交流模式中，写手是主要对话对象；圆桌模式中，写手被压缩进稿纸区域，同时议员在下方展开讨论。

## 影响

这个决策会让共享数据更清楚：

- 一份正文
- 一份小说材料
- 一个写手身份
- 一套消息记录模型
- 两种模式只负责不同展示和操作

它也能让后续多 AI 或多 agent 协作时边界更清楚：

- UI 优化主要碰渲染器和面板。
- 功能开发主要碰领域服务和控制器。
- 提示词和上下文优化主要碰上下文构建器。

代价是：需要逐步把 `src/main.js` 里的逻辑抽出去。这个过程必须小步推进，避免破坏当前已经被接受的移动端 UI。

## 考虑过的替代方案

### 把交流模式和圆桌模式做成两个独立应用

局部看会更简单，但会重复存储、写手设置、提示词逻辑和正文同步逻辑。

不采用。因为用户期望两种模式属于同一个创作房间。

### 完整重写成前端框架

这可能带来更干净的组件边界，但会拖慢当前产品推进，也有可能丢掉已经被接受的移动端 UI。

暂时不采用。

### 继续保持当前 `main.js` 集中结构

短期最快，但每加一个功能都会增加破坏其他功能或 UI 的概率。

不作为长期方向。过渡期可以继续保留，但要逐步抽离。

## 后续工作

- 定义参与者辅助模型。
- 定义消息记录辅助模型。
- 把上下文构建移到专门服务。
- 抽出交流模式控制器。
- 抽出圆桌模式控制器。
- 为主要移动端界面建立手动回归检查清单。
