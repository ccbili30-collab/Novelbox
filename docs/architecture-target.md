# TBird 目标架构

更新时间：2026-05-12

## 产品模型

TBird 正在从单纯的 Chatbox 式聊天工具，转向一个以“创作会话”为核心的小说工作间。

用户能看见的产品结构有三层：

1. 历史会话
2. 单个创作会话
3. 创作会话里的两种工作模式

这两种模式不是两个独立产品，而是同一个创作会话的两种视图：

- 交流模式：用户主要和写手直接创作。
- 圆桌模式：写手变成上方的稿纸正文，下方展开议员讨论。

用产品语言说：

> 交流模式是写手占据整个舞台。圆桌模式是同一个写手被压缩进稿纸，会议在稿纸下方展开。

## 目标结构

```text
历史会话
  -> 创作会话 Session
      -> 小说资产 Novel Assets
      -> 参与者 Participants
      -> 消息记录 Transcript
      -> 模式状态 Mode State
          -> 交流模式
          -> 圆桌模式
```

## 创作会话

创作会话是 TBird 的最高级创作单位。

一个创作会话拥有：

- 标题
- 消息历史
- 小说资产
- 写手设置
- 议员列表
- 当前模式状态
- 外观设置

创作会话不应该只被当成一条聊天记录。它应该被理解成一个完整的写作房间。

## 小说资产

小说资产由两种模式共享：

- 正文
- 剧情线
- 角色卡
- 世界观
- 大纲
- 伏笔
- 导入文本

不同模式可以用不同方式展示这些资产，但底层数据应该是同一份。

例如：

- 交流模式可以把正文库作为工具或面板展示。
- 圆桌模式可以把正文显示成上方漂浮稿纸。
- AI 上下文默认应该读取裁剪后的正文片段，而不是把完整正文全部塞进每次请求。

## 参与者

参与者是创作会话里的“人”或“AI 代理”。

目标参与者类型：

- 用户
- 写手
- 议员
- 系统

写手不是普通议员。写手是这个会话里主要负责生成正文的创作代理。

议员是讨论代理。议员可以批评、审批、提问、反驳、建议方向。默认情况下，议员应该短句发言，不应该擅自输出可直接放进小说正文的长段文本，除非用户明确要求。

每个 AI 参与者后续应该支持：

- 显示名称
- 角色类型
- 头像
- 系统提示词
- 激活档案
- API 设置
- 模型
- 温度
- 可读取材料设置

AI 参与者的 API 设置默认继承全局 API。用户需要不同类型模型一起讨论时，可以给单个参与者单独覆盖 API 或模型。

## 消息记录

消息记录是一个创作会话里实际发生过的内容。

每条消息后续应该明确记录：

- 消息 id
- 说话者 id
- 说话者类型
- 所属模式
- 内容
- 消息类型
- 创建时间
- 分支或父子关系

消息类型应该区分：

- 普通聊天气泡
- 写手正文输出
- 议员评论
- 系统事件
- 用户编辑
- 重新生成分支

这样可以避免把“谁说的话”和“这句话应该长什么样”混在一起。

## 两种模式

模式状态决定同一个创作会话如何被展示和操作。

### 交流模式

交流模式优化的是用户与写手的直接创作。

它应该展示：

- 用户消息
- 写手回复
- 写手设置
- 小说工具
- 紧凑的正文库入口

交流模式不应该维护另一套写手身份。它应该调用圆桌模式同一个写手参与者。

### 圆桌模式

圆桌模式优化的是会议式共创。

它应该展示：

- 靠近顶部的漂浮稿纸
- 稿纸下方的群聊式讨论
- 有顺序的议员发言
- @ 选择器
- 靠近输入框的开始/结束本轮按钮
- 作为底部弹层的参会人设置

写手通过稿纸和写手消息出现。议员通过讨论时间线出现。

## 上下文构建

AI 上下文构建应该是领域服务，不应该散落在 UI 事件里。

上下文构建器应该接收：

- 目标参与者
- 当前创作会话
- 选中的消息范围
- 材料读取权限
- 当前模式
- 用户指令

它应该输出：

- 系统消息
- 角色或人格信息
- 被选中的聊天历史
- 被选中的小说材料
- 正文片段
- @ 信息

这样可以让“某个 AI 到底能读什么”变得明确、可测试、可调整。

## 控制器边界

当前 `src/main.js` 后续应该逐步变成应用入口和协调层，而不是所有行为的所有者。

目标控制器：

- 会话控制器：创建、切换、重命名、删除、分支
- 交流模式控制器：用户与写手直接对话
- 圆桌控制器：议员顺序、@、写手接续
- 参与者控制器：编辑、激活、API/模型设置
- 正文控制器：导入、导出、同步、追加、可见片段
- 面板控制器：弹层、设置、抽屉、对话框

控制器可以调用领域服务和 UI 渲染器，但不应该拥有和自己无关的状态。

## 渲染边界

渲染应该按界面区域拆分：

- 历史会话列表
- 交流模式消息流
- 圆桌模式消息流
- 漂浮稿纸
- 参会人设置
- 设置弹窗
- 输入框

渲染器只负责把数据变成界面。渲染器不应该决定 AI 提示词、API 设置或议员发言顺序。

## 重构顺序

重构必须小步推进。

1. 先写清目标架构。
2. 定义参与者和消息记录的辅助模型。
3. 把上下文构建从 UI 事件里移出来。
4. 抽出交流模式控制器。
5. 抽出圆桌模式控制器。
6. 抽出参与者设置控制器。
7. 行为边界稳定后，再继续做 UI 细节和动效。

## 暂时不做

暂时不为了架构好看而引入后端服务。

暂时不把整个应用重写成前端框架，除非现有原生 JavaScript 结构已经成为明确阻碍。

暂时不让交流模式和圆桌模式使用两套互不兼容的存储格式。

暂时不让议员默认输出长篇正文。

## 验证规则

每次重构都应该保持以下检查通过：

```powershell
node --check src\main.js
node --check dev-server.mjs
```

涉及 UI 的重要变化，要以 Android WebView 真机效果为准，不能只看桌面浏览器预览。
