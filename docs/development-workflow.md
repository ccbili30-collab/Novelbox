# TBird 开发分工与优化流程

更新时间：2026-05-12

## 基本原则

TBird 现在按“创作会话优先”的方向推进。任何优化都应该先判断它属于哪一类：

- UI 体验
- 功能行为
- 数据模型
- AI 上下文
- Android 兼容
- 文档和交接

不要把这些问题混在一次改动里。

## 推荐分工

### 用户负责

用户负责产品判断和验收：

- 描述 bug 或新功能
- 标注优先级
- 判断 UI 是否符合“干净、整洁、排除杂念、行动丝滑”
- 在手机上确认真实体验
- 决定某个方案是否采纳

用户不需要写技术方案。

### Codex 负责

Codex 负责工程落地：

- 拆分任务边界
- 判断改动属于 UI、功能、状态、提示词还是 Android
- 编写审批稿或架构说明
- 修改代码
- 运行检查
- 打包 APK
- 真机验证
- 提交 Git 检查点
- 写清楚后续风险

## Bug 与新功能表建议格式

用户可以按这个格式整理需求：

| 类型 | 位置 | 当前表现 | 期望表现 | 优先级 | 截图/备注 |
| --- | --- | --- | --- | --- | --- |
| bug | 圆桌模式/稿纸 | 拖动到 0% 会跳变 | 可以丝滑折叠 | 高 | 手机截图 |
| 功能 | 参会人设置 | 议员不能单独设模型 | 可继承全局，也可覆盖 | 中 | 需要模型列表 |
| UI | 普通交流气泡 | 不够贴合文字 | 像圆桌气泡一样自然 | 高 | 对照圆桌 |

## Codex 独立优化范围

用户整理表格期间，Codex 可以独立做这些事：

- 抽离纯常量和纯函数
- 补充目标架构文档
- 建立 ADR
- 拆分 `src/main.js` 中不改变行为的领域模型
- 给现有功能补验证清单
- 清理临时调试文件
- 运行语法检查

Codex 不应独立做这些事：

- 改 UI 审美方向
- 改按钮位置和主流程
- 删除用户能看见的功能
- 改 AI 默认人格策略
- 修改 Android 权限或后台行为
- 合并多个互相影响的大功能

## 重构节奏

每一轮重构只做一个明确目标。

推荐顺序：

1. 领域模型抽离
2. 上下文构建抽离
3. 交流模式控制器抽离
4. 圆桌模式控制器抽离
5. 参与者设置控制器抽离
6. UI 局部优化
7. Android 真机回归

## 验证要求

每次代码改动至少运行：

```powershell
node --check src\main.js
node --check dev-server.mjs
```

新增模块时，也要单独检查新增模块：

```powershell
node --check src\domain\roundtable\roundtable-model.js
```

涉及 Android 或移动端视觉时，必须打包并真机查看，不能只依赖桌面浏览器。

## Git 规则

重要节点必须提交成检查点。

推荐分支：

- `codex/current-ui-baseline-*`：用户认可的 UI 基线
- `codex/architecture-refactor-*`：架构重构
- `codex/ui-polish-*`：纯 UI 优化
- `codex/feature-*`：功能开发

不要把临时截图、调试文件、个人工具配置混进产品提交。
